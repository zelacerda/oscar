# Workflow

## TDD Policy

**Moderado** — testes são encorajados e esperados, mas não bloqueiam a implementação. Lógica de negócio crítica (como cálculo de pontuação) deve ter cobertura de testes.

## Commit Strategy

**Conventional Commits** — seguir o padrão:

- `feat:` — nova funcionalidade
- `fix:` — correção de bug
- `chore:` — tarefas de manutenção
- `docs:` — documentação
- `test:` — adição/alteração de testes
- `refactor:` — refatoração sem mudança de comportamento
- `style:` — formatação, sem mudança de lógica

## Code Review

**Opcional / self-review OK** — o desenvolvedor pode aprovar suas próprias mudanças.

## Verification Checkpoints

**Apenas na conclusão do track** — verificação manual é exigida somente quando um track inteiro é concluído.

## Task Lifecycle

1. **Todo** — task criada e aguardando início
2. **In Progress** — desenvolvimento ativo
3. **Review** — aguardando revisão (se aplicável)
4. **Done** — concluída e verificada
